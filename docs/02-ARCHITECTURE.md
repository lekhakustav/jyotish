# 02 — Architecture

## Stack
- **SwiftUI**, iOS 17+, single target `Jyotish`, generated by **XcodeGen** (`project.yml`).
- No third-party packages. No network in v1. Everything computed and stored on device.
- Build: `xcodegen generate && xcodebuild -scheme Jyotish -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build`

## State
- `AppState: ObservableObject` — single source of truth injected as `@EnvironmentObject`.
  Holds: `session` (logged-out / loggedIn(Profile)), `family: [FamilyMember]`,
  `events: [PatroEvent]`, `chat: [ChatMessage]`, `language: Language`, `theme: ThemeChoice`.
- Every mutation goes through `AppState` methods which persist via `DataStore`.

## Supabase-readiness (the whole point of the service layer)
v1 ships **local-only**, but every I/O path goes through protocols so wiring Supabase later
is a one-file change:

```swift
protocol AuthService  { func signIn(...) async throws -> UserAccount; func signOut() ... }
protocol DataStore    { func loadProfile() / saveProfile / loadFamily / saveFamily
                        loadEvents / saveEvents / loadChat / saveChat }
```
- `DummyAuthService` — instant fake account (the "Continue" button).
- `LocalDataStore` — JSON files in `Documents/` (atomic writes, Codable).
- `SupabaseAuthService` / `SupabaseDataStore` — compile-ready stubs with `// TODO(supabase)`
  markers and a `SupabaseConfig` struct (url + anonKey placeholders). Table schema documented
  in SupabaseStubs.swift: `profiles`, `family_members`, `events`, `chat_messages` —
  all keyed by `user_id`, mirroring the Codable models 1:1.

## Profile gating
Kundali-dependent features (rashifal personalization, kundali screen, chat questions that
need a chart) check `profile.hasBirthData`. If missing → `ProfileGateView` prompt:
"Please complete the birth profile first" with a button straight to ProfileSetup.

## Persistence formats
Single `household.json` per account: `{ profile, family, events, chatByMember }`,
versioned (`schemaVersion: 1`) for future migration.
