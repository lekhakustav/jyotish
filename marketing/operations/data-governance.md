# Marketing data governance

## Source-of-truth layers

| Layer | Purpose | Mutation rule |
| --- | --- | --- |
| Platform export in Drive | Original aggregate export | Append only; never overwrite |
| `data/raw/` | Small privacy-safe raw snapshots or export metadata | Append only; correction gets a new ingestion ID |
| `data/normalized/` | Stable cross-platform row contracts | Regenerated by versioned transformation code |
| `data/derived/` | Rates, cohorts, confidence intervals, report inputs | Regenerated; never hand-edited |
| `analytics/reports/` | Dated interpretation and decisions | New report or explicit amendment |

Git is an audit trail and collaboration layer, not a user-event warehouse. Supabase or a later
restricted warehouse remains the home for user-level product events. Google Drive remains the
home for large platform exports and all binary media.

## Allowed aggregate data

- Daily platform ad delivery by platform ad/publication ID.
- Cumulative organic post snapshots at a recorded observation time.
- Aggregate acquisition and retention cohorts with no user or install identifiers.
- Aggregate product event counts grouped only by approved acquisition and app dimensions.
- Currency conversion reference rates with retrieval source and date.

## Never store in marketing Git or Drive

- `user_id`, `install_id`, `session_id`, device identifiers, advertising identifiers, IP
  addresses, authentication tokens, or account credentials.
- Names, email addresses, phone numbers, birth details, family relationships, coordinates,
  Kundali payloads, QR payloads, calendar content, notification text, voice transcripts, or chat
  text.
- Raw comments with usernames or profile links. Theme-code qualitative feedback and retain only
  aggregate counts or privacy-safe paraphrases.
- Inferred religion, caste, health, finances, relationship status, or vulnerability.

Git history is difficult to erase. If prohibited information is found, stop ingestion, notify the
repository owner, rotate exposed credentials if relevant, and follow an approved history-removal
and incident process. Do not merely delete the current working-tree file and assume the data is
gone.

## Raw export protocol

Every export receives an `ingestion_id` before normalization. Add a row to
`registry/ingestion-runs.csv` containing:

- platform, export type, account time zone, coverage period, and export timestamp;
- click and view attribution windows;
- Drive/local source path, SHA-256, source row count, and source schema version;
- parser version, import timestamp, and validation status.

Never edit an already tracked file under `data/raw/`. Platform restatements and corrected exports
receive a new file and ingestion row. Normalized rows keep `source_export_id` so every number can
be traced back.

## Null, zero, and corrections

- Blank/null means unavailable or not reported.
- Zero means the source explicitly reported zero.
- Negative source delivery counts are invalid.
- Negative derived daily deltas may be valid when a platform restates a cumulative snapshot.
- Refund and accounting adjustment facts may be negative only in schemas that explicitly permit
  them.
- Do not clamp corrections or silently carry forward missing observations.

## Time and currency

Store event/export timestamps in UTC ISO 8601. Preserve the source account time zone and the
platform-local reporting date. Store money in the original ISO-4217 currency as decimal text;
derive converted values using a dated rate from `data/reference/fx-rates.csv` and disclose the
rate source in the report.

## Schema evolution

Every registry and fact row contains a schema version. Additive optional fields may increment a
minor documentation revision; renamed fields, meaning changes, or grain changes require a new
schema version and explicit migration. Never change the meaning of an existing metric key.

Run `npm run marketing:validate` before committing. A pass establishes structural consistency and
basic privacy hygiene; it does not prove source accuracy, causal validity, or legal compliance.
